# `FlagState` rows are created eagerly for every environment, not lazily

When a flag is created, a `FlagState` row is inserted immediately for every environment that exists in the project at that moment, all defaulting to `status: inactive`. The alternative — creating a `FlagState` row lazily on first toggle — was rejected because it would force every downstream query (snapshot, SSE, dashboard status table) to handle a missing row as an implicit "inactive" state; eager initialization means a `FlagState` for `(flag, environment)` always exists for environments present at flag-creation time, so those code paths don't need a null-state fallback. This decision does not by itself cover environments created _after_ a flag already exists — see the acceptance criteria of the environment-creation issue for whether that backfill happens.
