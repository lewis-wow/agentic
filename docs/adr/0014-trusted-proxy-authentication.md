# Authentication is delegated to an operator-supplied reverse proxy, not built in

> **Superseded by [ADR-0024](0024-jwt-verified-trusted-proxy-identity.md)** for how the proxy's assertion is trusted: the shared-secret + plain-header mechanism described below was replaced by signature-verified JWT assertions, and signature-verifying proxies are no longer out of scope — they're now the only supported mechanism. The rest of this ADR (delegating auth to a reverse proxy at all, rather than building login/passwords/sessions) still holds.

The platform has no login page, password, or session of its own. The operator deploys `apps/dashboard`/`apps/bff` behind a reverse proxy (oauth2-proxy, Authelia, Pomerium) that authenticates the user and asserts their identity via a header; a shared Trusted Proxy Secret header guards against a client reaching the app directly and forging that identity. This replaced an earlier `better-auth` (email + password) implementation: for a self-hosted internal tool, operators already run an auth proxy in front of their infrastructure, so building and maintaining a second credential store and login flow duplicates that investment. Signature-verifying proxies (Cloudflare Access, GCP IAP, AWS ALB) are explicitly out of scope for v1 — only plain-header proxies are supported.
