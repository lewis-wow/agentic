# SSE reconnect replay is durable, backed by `FlagState`/`FlagDeletion`, not a bounded in-memory buffer

Reconnecting SDK clients (via `Last-Event-ID`) used to replay from a per-project in-memory ring buffer capped at 500 raw events, evicted oldest-first. This had two failure modes: a single flag changing frequently could fill the entire cap with its own history and push other flags' events out of the window before a reconnecting client caught up, and any `apps/api` restart wiped the buffer entirely, forcing every reconnect into a full snapshot regardless of how small the actual missed change was. We replaced it with a durable design: `FlagState` gains an `eventId` column, bumped in the same transaction as any change to `status`/`type`/`rollout`/`rules`, since the current row already _is_ the latest event's value (the same insight behind [ADR-0013](./0013-archive-is-state-level.md): archived is a `status` value, not a separate signal) — this covers created/updated/archived/unarchived with no separate cache. Deletion is the one case a mutating column can't represent, since the row disappears with the flag; a minimal `FlagDeletion` tombstone (`projectId`, `key`, `id`) records that a key is gone. `FlagState.eventId` and `FlagDeletion.id` both draw from one shared Postgres sequence so ids stay comparable for merge-and-sort replay across the two sources. Reconnect replay is now "query everything newer than `Last-Event-ID`," always complete, with no eviction-driven staleness case to detect. `FlagDeletion` rows are never pruned — growth is bounded by the number of flags ever deleted over an installation's lifetime, which for the single-tenant, self-hosted deployment described in [ADR-0003](./0003-sse-broadcast-in-memory-emitter.md) is negligible. A "safe to prune once every client has caught up" mechanism would need per-connection offset tracking (the same problem Kafka consumer groups solve for log retention) — disproportionate complexity for a bound that's already small enough not to matter.
