# The flag event bus and durable-replay logic live in `packages/api`, not `apps/api`

The flag event emitter moves from `apps/api/src/events/` into `packages/api`, exposed via a new `@repo/api/events` subpath, splitting into two shapes rather than one. `flagEmitter` — the bare, dependency-free `EventEmitter` used for live in-process broadcast to already-connected SSE clients — stays a genuine module-level singleton: it has nothing to inject or swap, so constructor-injecting it the way services are ([see `packages/api/AGENTS.md`](../../packages/api/AGENTS.md)) would buy nothing. The durable read/write side introduced by [ADR-0020](./0020-durable-sse-replay-via-postgres.md) becomes a `FlagEventService`, constructed with `{ prisma }` in each route file exactly like `FlagService`/`ProjectService`/`SdkService` — safe to construct more than once since it holds no shared mutable state beyond the injected client. Both pieces move together because [`docs/specification/app-scoped-packages.md`](../specification/app-scoped-packages.md) already states the general rule — anything that doesn't need Hono belongs in `packages/api` — and nothing about the emitter actually depends on Hono; it had been called out as a deliberate exception ("app-specific runtime infrastructure a service shouldn't own directly"), which this decision overturns as inconsistent with the package's own stated boundary. The two shapes are kept distinct rather than forcing the whole thing through the constructor-injection convention: a plain broadcast bus with no external dependency has nothing to gain from DI, and constructing a separate instance per route file would silently fragment it into multiple disconnected buses, breaking delivery between the route that emits a change and the route that's listening for one.
