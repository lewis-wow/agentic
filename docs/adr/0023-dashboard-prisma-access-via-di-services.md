# apps/dashboard's few sanctioned Prisma touchpoints route through DI service classes, not raw queries

`apps/dashboard` has always had a small, fixed set of files allowed to import `@repo/prisma` directly — Trusted Proxy Authentication's identity resolution (`src/lib/guards.ts`) and the pre-project-existence setup bootstrap (`src/app/page.tsx`, `(dashboard)/layout.tsx`, `setup/**`) — since neither can go through the normal TanStack-Query-via-`apps/api` data path (one predates having a session; the other predates having a project). Until now those touchpoints called `prisma.user.upsert` / `prisma.project.count` / `prisma.project.create` inline, duplicating logic that already exists as constructor-DI service classes in `packages/api/src/services/` (e.g. `FlagService`), and with none of that layer's unit-test coverage. Two alternatives were rejected: routing these calls through new `apps/api` HTTP endpoints (an unnecessary network hop and JWT-claims change for logic that runs before a session or a project exists), and extracting a new `packages/dashboard` package (there's no second consumer and no framework-boundary reason to split, unlike the `apps/api`/`packages/api` Hono split). Instead, each touchpoint now goes through a DI service — `packages/bff`'s `UserService` and `packages/api`'s `ProjectService` (`exists()`, `createWithEnvironments()`) — constructed with the real `prisma` singleton and called in-process by `apps/dashboard`. This is the same shape as every other service in the repo: constructible and testable from a plain script or a test with zero awareness of Hono or Next.js. `packages/prisma/AGENTS.md` and `apps/api/AGENTS.md`'s "only `apps/api` may import `@repo/prisma`" language is relaxed accordingly — a caller may import the singleton solely to construct one of these DI services, never to query it directly.
