# Trusted Proxy Authentication verifies a signed JWT instead of trusting a shared secret and a plain header

Supersedes [ADR-0014](0014-trusted-proxy-authentication.md), which explicitly put signature-verifying proxies out of scope for v1. That scope is now in: `resolveTrustedProxyUser` no longer trusts a shared-secret header (`X-Trusted-Proxy-Secret`) plus a plain-text identity header (`X-Forwarded-Email`) blindly — anyone who could reach the app and reproduce those two header values could forge any identity, including the designated owner. Instead it cryptographically verifies a signed JWT assertion the reverse proxy sets (Pomerium's `X-Pomerium-Jwt-Assertion` and equivalents from any other JWT-capable proxy — GCP IAP, Cloudflare Access, AWS ALB all publish one), using [`jose`](https://github.com/panva/jose)'s `createRemoteJWKSet` + `jwtVerify` against the proxy's published JWKS endpoint. Verification requires: a signature valid under a key from the configured JWKS URL, `alg` matching an operator-configured allow-list (default `ES256`, Pomerium's default — never inferred from the token itself, to avoid algorithm-confusion), `iss` and `aud` matching operator-configured expected values (closing the token-confusion gap where a validly-signed token meant for a different application behind the same proxy could otherwise be replayed here), a non-expired `exp`, and a resolvable email at an operator-configured claim path (default `claims.email`, Pomerium's nested-array shape). Any failure — missing header, bad signature, disallowed algorithm, `iss`/`aud` mismatch, expired, unresolvable JWKS/kid, missing email claim — fails closed with 401; there is no fallback to the old header-trust path. `TRUSTED_PROXY_SECRET` and `TRUSTED_PROXY_IDENTITY_HEADER` are removed entirely rather than kept as defense-in-depth: once the signature itself proves the request came from the trusted proxy, a second weaker shared-secret check adds config surface without adding security. Providers that can't present a signed JWT (plain oauth2-proxy/Authelia header-only setups) are no longer directly supported — an operator on one of those puts a JWT-issuing proxy (e.g. Pomerium) in front instead, rather than the platform maintaining two trust models. The JWKS client is built once per process (its own in-memory key cache expects this) via a factory each app calls at startup and injects into the request-time verifier, matching the existing DI shape (`upsertUser` callback) rather than a module-level singleton — this also keeps it testable with a hand-written fake verifier per `docs/specification/testing.md`, with no `vi.mock` of `jose` or `fetch`. The umbrella term "Trusted Proxy Authentication" is unchanged — auth is still delegated to the proxy — but "Trusted Proxy Secret" and "Identity Header" are retired from the glossary in favor of "Proxy Identity JWT" and "JWKS Verification". `packages/auth/src/jwt.ts`'s hand-rolled RS256 signer/verifier is untouched by this change; it mints/verifies the BFF's own internal project-/SDK-scoped JWTs against a single static key this codebase controls, a different problem from verifying a third party's rotating, multi-algorithm key set.
