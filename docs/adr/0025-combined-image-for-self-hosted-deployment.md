# Self-hosted deployment can build a single combined image instead of five containers

For simple self-hosted deployment, `apps/api`, `apps/bff`, and `apps/dashboard` can be built into one combined image (`Dockerfile.selfhosted`) supervised by `tini` plus a custom entrypoint script that spawns all three as child processes on their existing dev-convention ports (api `3001`, bff `3002`, dashboard `3000`), forwards signals, and exits the whole container if any one process dies. Postgres stays a separate container — bundling a stateful database engine into the same image as stateless app processes was rejected as fighting Docker's one-process-per-container model and complicating independent version upgrades and backups. Only `dashboard` and `bff` are published from this image; `api` remains bound to localhost only, since nothing outside the stack ever talks to it directly (dashboard and SDK clients go through `bff`, per [ADR-0001](0001-sdk-routes-through-bff.md)).

This is purely a packaging decision. It does not bundle a reverse proxy — Trusted Proxy Authentication ([ADR-0014](0014-trusted-proxy-authentication.md), [ADR-0024](0024-jwt-verified-trusted-proxy-identity.md)) is still entirely bring-your-own, unchanged from the multi-container setup. It does not change `NEXT_PUBLIC_APP_URL`'s Docker-build-arg mechanism ([ADR-0017](0017-next-public-app-url-is-a-build-arg.md)) — there is no single prebuilt image published to a registry; self-hosters build their own with their own URL baked in. It does not consolidate the three apps' separate `dotenvx` private keys/`.env.production` files — each child process still loads its own, unchanged.
