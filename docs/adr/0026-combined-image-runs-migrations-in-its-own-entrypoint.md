# The combined self-hosted image runs migrations synchronously in its own entrypoint, not a dedicated container

[ADR-0016](0016-dedicated-migrator-container.md) runs migrations in a dedicated one-shot container so migration failure is an explicit, independently observable precondition via Compose's `service_completed_successfully` — but the combined image ([ADR-0025](0025-combined-image-for-self-hosted-deployment.md)) has no separate container to put that step in. Its entrypoint instead polls `DATABASE_URL` until Postgres accepts connections (bounded timeout, then fails loudly rather than hanging forever), runs `prisma migrate deploy` synchronously as the very next step before spawning `api`/`bff`/`dashboard`, and exits non-zero without starting any app process if either step fails. This preserves ADR-0016's intent — migration failure blocks the app rather than being silently swallowed inside one app's own boot sequence — even though the mechanism (entrypoint log ordering and exit code) differs from Compose's dependency graph.

This exception applies only to the combined image. The regular multi-container `docker-compose.yml` is unchanged and still uses the dedicated `migrator` container.
