# The combined image runs migrations synchronously in its own entrypoint, not a dedicated container

> **Superseded by [ADR-0027](0027-wait-for-postgres-then-single-migrate-attempt.md)** for _how_ the entrypoint waits out a cold-starting Postgres: retrying `prisma migrate deploy` itself was replaced by a `pg_isready` readiness wait followed by a single migrate attempt, so "Postgres not reachable yet" and "migration failed" are distinguishable in the logs. Running migration synchronously in the entrypoint before any app process starts, and exiting non-zero without starting anything if it doesn't succeed, still holds.

[ADR-0016](0016-dedicated-migrator-container.md) runs migrations in a dedicated one-shot container so migration failure is an explicit, independently observable precondition via Compose's `service_completed_successfully` — but the combined image ([ADR-0025](0025-combined-image-for-self-hosted-deployment.md)) has no separate container to put that step in. Its entrypoint instead retries `prisma migrate deploy` itself in a bounded loop (fixed attempt count, fixed delay between attempts) before spawning `api`/`bff`/`dashboard`, and exits non-zero without starting any app process if it never succeeds. One retry loop deliberately covers both failure shapes — "Postgres isn't reachable yet" on a cold start, and "the migration itself is broken" — rather than a separate raw DB-reachability probe followed by a single migrate attempt: both must resolve to the same observable outcome (proceed once migrated, or fail loudly within a bounded window), so one mechanism is simpler than two. This preserves ADR-0016's intent — migration failure blocks the app rather than being silently swallowed inside one app's own boot sequence — even though the mechanism (entrypoint log ordering and exit code) differs from Compose's dependency graph.

This exception applies only to the combined image. The regular multi-container `docker-compose.yml` is unchanged and still uses the dedicated `migrator` container.
