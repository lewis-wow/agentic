# s6-overlay supervises the combined image's processes, not tini + a hand-rolled bash entrypoint

Supersedes the supervision mechanism described in [ADR-0025](0025-combined-image-for-self-hosted-deployment.md) and the entrypoint implementation in [ADR-0026](0026-combined-image-runs-migrations-in-its-own-entrypoint.md)/[ADR-0027](0027-wait-for-postgres-then-single-migrate-attempt.md). All three ADRs' actual decisions still hold — one combined image, migration as a synchronous hard precondition, `pg_isready` then a single `prisma migrate deploy` attempt — only the tool executing them changes.

The previous `docker/selfhosted-entrypoint.sh` hand-rolled everything s6-overlay gives for free: a `spawn_child` function to isolate each app's `DOTENV_PRIVATE_KEY`/port env vars in a subshell, a `trap ... TERM INT` plus manual `kill -TERM` fan-out for signal forwarding, and `wait -n "${child_pids[@]}"` followed by killing the rest to make any one process's exit bring down the whole container. That's a plausible amount of process-supervision logic to keep re-deriving and re-testing by hand as the image's needs grow.

s6-overlay (`docker/selfhosted-rootfs/etc/s6-overlay/`) replaces it with declarative service definitions: `migrate` is a oneshot (`s6-rc.d/migrate/up` invokes `scripts/migrate-wait.sh` — the same `pg_isready` loop + single migrate attempt as before, just as its own file instead of inline in the entrypoint); `api`/`bff`/`dashboard` are longruns, each declaring `migrate` as a `dependencies.d` entry so s6-rc won't start any of them until migration succeeds, and each with a `finish` script that calls `/run/s6/basedir/bin/halt` — the equivalent of the old `wait -n` fan-out, making any one app process's exit (crash or clean) bring down the whole container rather than leaving the other two running against a half-dead stack. `S6_BEHAVIOUR_IF_STAGE2_FAILS=2` is what makes a failed `migrate` oneshot halt the container instead of s6-overlay's default of continuing regardless — the direct equivalent of the old entrypoint's `exit 1` before spawning any app.

One real behavior change: s6-overlay's PID 1 (`/init`) needs root to supervise the process tree, so this image no longer runs the whole container as `USER nodejs` the way the tini version did. Each service's `run`/`up` script drops to `nodejs` itself via `s6-setuidgid` before exec'ing its actual process (`node`, `dotenvx`, `prisma`), so the app processes are still unprivileged — only s6-overlay's supervisor itself runs as root.

`docker/selfhosted-entrypoint.sh` is removed entirely; nothing else references it (`Dockerfile.selfhosted` copies `docker/selfhosted-rootfs/` into the image root instead of a single entrypoint script, and swaps `ENTRYPOINT ["tini", "--"]` / `CMD ["./entrypoint.sh"]` for `ENTRYPOINT ["/init"]`). This exception still applies only to the combined image — the regular multi-container `docker-compose.yml` is unchanged and still uses the dedicated `migrator` container ([ADR-0016](0016-dedicated-migrator-container.md)).
