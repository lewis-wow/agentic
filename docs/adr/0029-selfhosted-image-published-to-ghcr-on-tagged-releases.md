# The self-hosted image is published to GHCR on tagged releases, not build-your-own only

Supersedes one specific line of [ADR-0025](0025-combined-image-for-self-hosted-deployment.md): "there is no single prebuilt image published to a registry; self-hosters build their own with their own URL baked in." Every other decision in ADR-0025 (one combined image, Postgres as a separate container, only `dashboard`/`bff` reachable from outside the image) and all of [ADR-0028](0028-s6-overlay-supervises-the-combined-image.md)'s supervision mechanism are unchanged.

Pushing a git tag matching `v*.*.*` now runs a release pipeline (the repo's first CI/CD pipeline) that builds `Dockerfile.selfhosted`, smoke-tests the resulting image against a real Postgres instance via `scripts/smoke-test-selfhosted.sh`, and — only if that passes — publishes it to GitHub Container Registry as `ghcr.io/lewis-wow/agentic-selfhosted`, tagged with both the exact tag string and a floating `latest`. The package is public: anyone can `docker pull` it without authenticating. A GitHub Release is created alongside it with generated notes.

The one open problem ADR-0025 already named — `NEXT_PUBLIC_APP_URL` ([ADR-0017](0017-next-public-app-url-is-a-build-arg.md)) is baked into the dashboard's client bundle at `next build` time, and is inherently deployment-specific — isn't solved here, just resolved pragmatically: the published image's build never supplies that build arg, so it ships with it unset (relative/same-origin behavior). This covers the common case of a self-hoster reverse-proxying the whole stack on one origin. A self-hoster who needs a different public URL still builds their own image locally with their own value, exactly as ADR-0025 originally described — that escape hatch is unchanged, it's just no longer the only path.

GHCR packages default to private even when the owning repository is public, and flipping that to public requires either a one-time manual toggle in the package's own settings or a personal access token with `admin:packages` scope — the workflow's built-in `GITHUB_TOKEN` can push images (`packages: write`) but cannot change a package's visibility. The first release's package therefore needs that one manual toggle after its first publish; every subsequent tag push publishes new versions under the now-public package with no further action.
